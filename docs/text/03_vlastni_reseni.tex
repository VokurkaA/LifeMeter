\section{Architektura systému}

Systém LifeMeter se skládá ze tří hlavních součástí: mobilního klienta, webové
vrstvy a~serverové části nasazené v~cloudovém prostředí. Mobilní i~webový klient
komunikují se serverem za použití REST API přes protokol HTTPS. Celkové uspořádání systému je znázorněno
na obrázku~\ref{fig:system-overview}.

\subsection{Klientská vrstva}

Mobilní aplikace je vyvinuta pomocí frameworku React Native ve spojení s~platformou
Expo~\cite{expo}, která zajišťuje přenositelnost mezi operačními systémy Android a~iOS
z~jedné sdílené kódové základny. Na platformě Android jsou zdravotní data čtena přes
rozhraní Health Connect~\cite{healthconnect2024}, na platformě iOS pak prostřednictvím
HealthKit~\cite{healthkit2024}.

Aplikační stav je spravován čtveřicí doménových Context Providers -- \texttt{SleepStore}\allowbreak\texttt{Context},
\texttt{Nutrition}\allowbreak\texttt{StoreContext}, \texttt{Workout}\allowbreak\texttt{StoreContext} a~\texttt{UserStore}\allowbreak\texttt{Context}.
Každý provider zapouzdřuje data a~operace pro příslušnou doménu a~vystavuje je potomkům
stromu komponent prostřednictvím React Context API. Lokální perzistenci dat zajišťuje knihovna
MMKV~\cite{mmkv2024}: data jsou primárně čtena z~cache v~paměti zařízení a~synchronizována
se serverem při obnovení síťového připojení, v~souladu s~principem \textit{offline-first}~\cite{offlinefirst2015}.
Přihlašovací údaje jsou ukládány šifrovaně prostřednictvím \texttt{expo-secure-store}.

\subsection{Webová vrstva}

{\sloppy
Webová část systému je implementována v~adresáři \texttt{Web/} pomocí
frameworku Next.js a~slouží dvěma odlišným účelům. Veřejné routy
\texttt{/} a~\texttt{/downloads} fungují jako landing page a~download
centrum verzovaných mobilních buildů; jsou generovány staticky
s~krátkou periodou revalidace, aby bylo možné rychle obsloužit běžné
návštěvy a~současně promítnout nově publikované buildy. Druhou část
tvoří administrátorská konzole pouze pro čtení pod cestou \texttt{/admin},
která po ověření role \texttt{admin} zobrazuje systémový přehled,
logy a~detail uživatelů. Route handlery
\texttt{/downloads/[platform]/latest} a~
\texttt{/downloads/[platform]/[fileName]} zpřístupňují aktuální i~
archivní instalační soubory přímo z~adresářů připojených do webového
kontejneru.
}

\subsection{Serverová vrstva}

Serverová část je implementována v~prostředí Bun s~využitím webového frameworku
Hono~\cite{hono2024}. Každý příchozí požadavek prochází globálním middlewarovým řetězcem:
\texttt{requestLogger} zaznamenává HTTP provoz, autentizační middleware injektuje informace
o~relaci (\texttt{c.get("user")}) a~\texttt{pagination()} normalizuje parametry stránkování.
Správu uživatelských relací zajišťuje knihovna Better Auth~\cite{betterauth2024} prostřednictvím
stavových \texttt{HttpOnly} cookies, v~souladu s~doporučeními OWASP~\cite{owasp2024mobile}.

Routy jsou organizovány doménově: \texttt{/api/auth} zpracovává přihlášení a~registraci,
\texttt{/api/food} poskytuje přístup k~databázi potravin, \texttt{/api/user/*} sdružuje
endpointy pro spánek, jídla, tréninky, uživatelský profil a~synchronizaci
zdravotních dat, \texttt{/api/workout} vystavuje referenční data (cviky, styly sérií),
\texttt{/api/logs} systémové logy a~\texttt{/api/admin} administrátorské přehledy nad
uživatelskými daty. Chráněné routy vyžadují middleware \texttt{requireAuth()},
administrátorské pak \texttt{requireAdmin()}.

\subsection{Datová vrstva}

Datová vrstva je tvořena relační databází PostgreSQL~17~\cite{postgresql2024}.
Rozšíření \texttt{pg\_trgm}~\cite{pg_trgm2024} umožňuje efektivní trigram-based
fulltextové vyhledávání v~databázi potravin, jejíž obsah pochází z~veřejně dostupného
zdroje USDA FoodData Central~\cite{usda2024}. Serverová část komunikuje s~databází
prostřednictvím connection poolu knihovny \texttt{pg}.

{\sloppy

Dále jsou generovány tabulky \texttt{session} (aktivní relace s~tokenem a~IP adresou, viz tabulka~\ref{app:tab:session}), \texttt{account} (přihlašovací metoda a~přihlašovací údaje, viz tabulka~\ref{app:tab:account}) a~\texttt{verification} (jednorázové ověřovací tokeny, viz tabulka~\ref{app:tab:verification}). \subsection{Výživa}
Skupina výživových tabulek se skládá ze dvou částí: referenčních dat importovaných z~databáze USDA FoodData Central~\cite{usda2024} a~uživatelských záznamů o~konzumovaných jídlech. Struktura těchto tabulek (\texttt{food}, \texttt{nutrient\_value}, \texttt{portion}, \texttt{user\_meal} a~\texttt{user\_food}) je detailně popsána v~příloze~B (tabulky~\ref{app:tab:food} až~\ref{app:tab:user_food}).

\subsection{Spánek}
Doména spánku zahrnuje tabulku \texttt{user\_sleep} pro záznam denního spánku uživatele (viz tabulka~\ref{app:tab:user_sleep} v~příloze~B).

\subsection{Tréninky}
Doména tréninků se skládá z~referenčních dat (typy cviků, svaly, styly a~typy sérií) a~uživatelských dat (šablony tréninků a~zaznamenané tréninky). Definice hlavních tréninkových tabulek (\texttt{workout\_template}, \texttt{template\_workout\_set}, \texttt{workout} a~\texttt{workout\_set}) jsou uvedeny v~příloze~B (tabulky~\ref{app:tab:workout_template} až~\ref{app:tab:workout_set}).

{\sloppy
Referenční tabulky \texttt{exercise} (kombinace typu a~varianty cviku), \texttt{exercise\_}\allowbreak\texttt{type}, \texttt{exercise\_}\allowbreak\texttt{variant}, \texttt{muscle} (hierarchická tabulka svalů s~vlastní rekurzivní vazbou \texttt{parent\_}\allowbreak\texttt{id}), \texttt{muscle\_}\allowbreak\texttt{involvement} (míra zapojení svalu v~cviku), \texttt{set\_}\allowbreak\texttt{style} a~\texttt{set\_}\allowbreak\texttt{type} obsahují pouze identifikátor a~název, případně doplňující číselné atributy, a~slouží jako číselníky (viz příloha~B).
}

\subsection{Uživatelský profil}
{\sloppy
Struktura profilu a~cílů je uložena v~tabulkách \texttt{user\_profile} a~\texttt{user\_goal} (viz tabulky~\ref{app:tab:user_profile} a~\ref{app:tab:user_goal} v~příloze~B).

Dále jsou evidovány tabulky \texttt{user\_weight\_log} (hmotnost, viz tabulka~\ref{app:tab:user_weight_log}), \texttt{user\_}\allowbreak\texttt{height\_}\allowbreak\texttt{log} (výška v~cm s~časovým razítkem), \texttt{user\_}\allowbreak\texttt{blood\_}\allowbreak\texttt{pressure\_}\allowbreak\texttt{log} (systolický a~diastolický tlak v~mmHg) a~\texttt{user\_}\allowbreak\texttt{heart\_}\allowbreak\texttt{rate\_}\allowbreak\texttt{log} (tepová frekvence v~bpm). Všechny protokolové tabulky sdílejí společný vzor: primární klíč \texttt{UUID}, cizí klíč \texttt{user\_id} s~\texttt{CASCADE DELETE} a~časové razítko \texttt{measured\_at} indexované sestupně pro efektivní výpis posledních hodnot. Kompletní definice těchto tabulek jsou k~dispozici v~příloze~B.
}

\subsection{Synchronizace zdravotních dat}

Doména synchronizace zdravotních dat je reprezentována trojicí tabulek.
Tabulka \texttt{user\_}\allowbreak\texttt{health\_}\allowbreak\texttt{sync\_}\allowbreak\texttt{state}
ukládá stav synchronizace pro jednotlivé providery (Health Connect
a~Apple Health). Tabulka
\texttt{user\_}\allowbreak\texttt{health\_}\allowbreak\texttt{sync\_}\allowbreak\texttt{pending\_}\allowbreak\texttt{run}
slouží k~rozpracovaným dávkovým běhům. Tabulka

mapuje zdrojové záznamy z~externích platforem na cílové řádky v~doménových

\texttt{user\_}\allowbreak\texttt{weight\_}\allowbreak\texttt{log},
\texttt{user\_}\allowbreak\texttt{height\_}\allowbreak\texttt{log},

Tato vrstva umožňuje idempotentní
dávkovou synchronizaci, detekci smazaných záznamů a~bezpečné obnovení

\section{Popis řešení}

\subsection{Mobilní aplikace}

\subsubsection{Autentizace a~správa relací}

{\sloppy
Autentizace je realizována prostřednictvím \texttt{AuthProvider} --
kontextu obalujícího celý strom aplikace. Při inicializaci se volá
\texttt{authService.\allowbreak getSession()}, jehož výsledek je uložen v~zabezpečeném
úložišti \texttt{expo-secure-}\allowbreak\texttt{store} pod klíčem \texttt{auth-session-}\allowbreak\texttt{cache}.
Při selhání síťového volání je relace obnovena z~tohoto lokálního záznamu,
čímž je zachována funkčnost i~při výpadku připojení. Přihlašování
a~registrace probíhá voláním REST endpointů Better Auth
(\texttt{/api/}\allowbreak\texttt{auth/}\allowbreak\texttt{sign-in/}\allowbreak\texttt{email}, \texttt{/api/}\allowbreak\texttt{auth/}\allowbreak\texttt{sign-up/}\allowbreak\texttt{email});
stavová \texttt{HttpOnly} cookie je spravována serverem a~automaticky
přikládána ke každému požadavku prostřednictvím volby \texttt{credentials:~"include"}.
}

\subsubsection{Modul zdravotních dat}

{\sloppy
Čtení zdravotních dat ze systémových úložišť je zapouzdřeno do abstrakční
vrstvy \texttt{src/lib/}\allowbreak\texttt{health/}, která sjednocuje rozhraní
Health Connect~\cite{healthconnect2024} na platformě Android
a~HealthKit~\cite{healthkit2024} na platformě iOS do jediného
platformově neutrálního API. Každá funkce (například \texttt{getSteps}, 
\texttt{getSleep}, \texttt{getWeight}) vrací strukturu
\texttt{Result<T>} rozlišující úspěšný výsledek a~typovanou chybu,
čímž je zamezeno nekontrolovanému šíření výjimek do vyšších vrstev.
}

Oprávnění jsou vyžádána jednorázově při startu aplikace funkcí
\texttt{requestHealth\-Permissions()}. Na platformě Android je inicializován
klient Health Connect a~pro každý typ dat (\texttt{Steps},
\texttt{SleepSession}, \texttt{Weight} aj.) je vyžádáno oprávnění ke čtení;
na platformě iOS je volána metoda \texttt{requestAuthorization()} knihovny
HealthKit. Hodnoty spánkových fází jsou mapovány z~číselných konstant
příslušné platformy na sjednocenou množinu
\texttt{"awake"\allowbreak\ | "core"\allowbreak\ | "deep"\allowbreak\ | "rem"\allowbreak\ | "asleep"}.

\subsubsection{Lokální úložiště}

Lokální perzistence dat je realizována modulem \texttt{src/lib/storage.ts},
který obaluje MMKV~\cite{mmkv2024} typově bezpečnými pomocnými funkcemi
pro čtení a~zápis primitiv, objektů a~polí serializovaných do JSON.
Pro použití uvnitř komponent React jsou exportovány hooky
\texttt{useStorage.string()}, \texttt{useStorage.boolean()},
\texttt{useStorage.object()} a~\texttt{useStorage.array()}, jejichž
změna hodnoty způsobí překreslení komponenty v~souladu s~modelem React.
Pro imperativní přístup mimo komponenty (například ve frontě offline
požadavků) slouží synchronní API objektu \texttt{storage}.

\subsubsection{Offline fronta požadavků}

Síťová vrstva aplikace (\texttt{src/lib/net.ts}) implementuje mechanismus
odložení mutujících požadavků při výpadku připojení. Při každém volání
funkce \texttt{request()} je nejprve zkontrolován příznak \texttt{isOffline}
ze sdíleného modulu \texttt{network-state.ts}. Je-li zařízení offline,
požadavek není odeslán, ale serializován a~přiložen do fronty uložené
v~MMKV pod klíčem \texttt{offline-queue}. Duplicitní záznamy jsou
odfiltrována porovnáním cesty a~těla požadavku.

{\sloppy
Stav sítě je monitorován komponentou \texttt{NetworkProvider}
prostřednictvím funkce \texttt{addNetworkState\-Listener()} z~knihovny
\texttt{expo-network}. Při obnovení připojení je zavolána funkce
\texttt{flushQueue()}, která sekvenčně přehraje uložené požadavky
a~neúspěšné ponechá ve frontě pro příští pokus. Souběžně je emitována
událost \texttt{emitReconnect()}, na kterou jsou přihlášeny všechny
doménové Context Providers a~reagují opětovným načtením dat ze serveru.
Podrobný popis algoritmů \texttt{request()} a~\texttt{flushQueue()} je
uveden v~sekci~\ref{sec:offline-fronta}.
}

\subsubsection{Lokální oznámení}

Plánování oznámení je centralizováno do hooku \texttt{useNotifications()}
v~souboru \texttt{src/lib/}\allowbreak\texttt{notifications.ts}. Funkce
\texttt{scheduleNotification()}
přijímá identifikátor, titulek, text, čas a~příznak opakování. Opakující
se oznámení (připomenutí spánku, vstávání, cvičení) jsou naplánována
s~triggerem typu \texttt{DAILY} zadáváním hodiny a~minuty; jednorázová
s~triggerem \texttt{TIME\_INTERVAL}. Stav naplánovaných oznámení je
synchronizován s~MMKV, aby byl dostupný i~bez živého systémového volání.
Při globálním vypnutí oznámení jsou zrušena všechna naplánovaná upozornění.

\subsection{Serverová část}

\subsubsection{Middlewarový řetězec}

Každý příchozí HTTP požadavek prochází middlewarovým řetězcem definovaným
v~\texttt{src/server.ts}. Jako první je aplikován middleware
\texttt{secureHeaders()} nastavující bezpečnostní HTTP hlavičky
(například \texttt{X-Content-Type-Options}, \texttt{X-Frame-Options}),
následuje \texttt{trimTrailingSlash()} pro normalizaci cest URL
a~\texttt{requestLogger}, který zaznamenává metodu, cestu, stavový kód,
dobu zpracování a~IP adresu klienta. Na všech cestách je poté injektována
informace o~relaci voláním \texttt{auth.api.getSession()}: výsledek
je uložen do kontextu jako \texttt{c.get("user")} a~\texttt{c.get("session")};
pro nepřihlášené požadavky jsou obě hodnoty nastaveny na \texttt{null}.
Chráněné routy uplatňují middleware \texttt{requireAuth()}, který vrací
odpověď \texttt{401~Unauthorized} při chybějící relaci, resp.\
\texttt{requireAdmin()}, jenž navíc ověřuje roli \texttt{"admin"} v~poli
role uživatele.

\subsubsection{Struktura endpointů}

Routy jsou organizovány do šesti doménových skupin pod základní cestou
\texttt{/api}. Skupina \texttt{/api/auth/*} je delegována přímo
na handler knihovny Better Auth a~nevyžaduje autentizaci. Skupina
\texttt{/api/workout/*} vystavuje veřejně dostupná referenční data
(cviky, styly a~typy sérií, váhové možnosti). Skupina
\texttt{/api/food} poskytuje stránkovaný seznam potravin a~fulltextové
vyhledávání (parametr \texttt{name} využívající \texttt{pg\_trgm}\allowbreak\ \cite{pg_trgm2024},
parametr \texttt{gtin} pro vyhledávání dle čárového kódu). Skupina
\texttt{/api/user/*} sdružuje chráněné endpointy pro spánek
(\texttt{/sleep}), výživu (\texttt{/food}), tréninky
(\texttt{/workout}, \texttt{/workout/template}), uživatelský profil
(\texttt{/data}) a~synchronizaci zdravotních dat (\texttt{/sync}).
Zbývající skupiny \texttt{/api/logs} a~\texttt{/api/admin/*} obsluhují
administrátorské logy a~dohled nad uživatelskými daty pouze pro čtení.
Schémata požadavků a~odpovědí jsou validována knihovnou Zod a~dokumentována
prostřednictvím integrace \texttt{@hono/zod-openapi}; výsledná OpenAPI
dokumentace je přístupná na \texttt{/api/doc} a~\texttt{/api/ui}.

\subsubsection{Nasazení a~reverzní proxy}

Serverová a~webová část jsou provozovány na instanci Oracle Cloud
Infrastructure s~architekturou ARM. Kontejnerizační soubor
\texttt{docker-\allowbreak compose.yml} definuje tři služby:
\texttt{db} (PostgreSQL~17 s~pojmenovaným svazkem \texttt{pgdata} pro
perzistenci dat), \texttt{api} (sestavení ze zdrojového kódu adresáře
\texttt{Server/}, mapování portu~3000 a~sdílený svazek pro výstup
logovacích souborů) a~\texttt{web} (Next.js frontend z~adresáře
\texttt{Web/}, publikovaný interně na portu~3001). Webový kontejner
má navíc mounty pouze pro čtení adresářů \texttt{releases/apk} a~
\texttt{releases/ios}, odkud obsluhuje aktuální i~archivní instalační
soubory pro download centrum. Docker Compose~\cite{docker2024} běží přímo
na VM; reverzní proxy Caddy je nainstalována odděleně jako systémová služba
a~spravuje ukončení TLS pro domény \texttt{lifemeter.fit}
a~\texttt{www.lifemeter.fit}.

Caddy je nakonfigurována se dvěma pravidly omezení frekvence požadavků
(\textit{rate limiting}) rozlišujícími dvě časová okna: maximálně
10~požadavků za sekundu a~100~požadavků za minutu, obojí klíčované
IP adresou klienta (\texttt{key \{remote\_host\}}). Požadavky na cestu
\texttt{/api} a~její podstromy jsou předávány na interní port~3000
aplikačního kontejneru, zatímco veškeré ostatní požadavky jsou směrovány
na webový frontend na portu~3001. Tím jsou veřejné stránky, download
centrum i~admin konzole obslouženy přes stejnou doménu jako samotné API.

\subsection{CI/CD pipeline}

{\sloppy
Průběžné nasazení serverové a~webové části je zajištěno workflow GitHub
Actions (\texttt{.github/workflows/deploy.yml}). Workflow se spouští
automaticky při každém přímém commitu nebo sloučení pull requestu do
větve \texttt{main}, pokud změna zasahuje do adresářů \texttt{Server/}
nebo \texttt{Web/}, případně do \texttt{docker-\allowbreak compose.yml}. Jediný krok
\textit{Deploy to Oracle VM} využívá akce
\texttt{appleboy/}\allowbreak\texttt{ssh-action}, která se připojí na produkční server
prostřednictvím SSH klíče uloženého v~tajných proměnných repozitáře
(\texttt{SSH\_HOST}, \allowbreak \texttt{SSH\_USER}, \allowbreak \texttt{SSH\_KEY}).
Na serveru jsou poté provedeny příkazy \texttt{git pull origin main},
\texttt{docker compose build --pull} (s~fallbackem na build bez cache při
selhání) a~\texttt{docker compose up -d --remove-orphans}, čímž dochází
k~nasazení aktualizované serverové i~webové vrstvy s~minimálním výpadkem.
Databáze zůstává v~provozu díky pojmenovanému svazku \texttt{pgdata}, který
není součástí životního cyklu kontejnerů.

Mobilní klient je publikován samostatným workflow
\texttt{.github/workflows/app-\allowbreak apk.yml}. Při změně adresáře
\texttt{App/} se v~GitHub Actions provede \texttt{npm ci},
\texttt{expo prebuild} pro Android, sestavení release APK pomocí
Gradle a~následné nahrání výsledného artefaktu na Oracle VM do adresáře
\texttt{releases/apk}. Současně jsou aktualizována metadata
\texttt{latest.json}, která využívá webová stránka \texttt{/downloads}
pro zobrazení nejnovějšího sestavení a~jeho verzovaného archivu.

\subsection{Diagram tříd}

Správa aplikačního stavu klientské vrstvy je realizována čtveřicí
doménových rozhraní (\textit{Context Providers}), z~nichž každý
zapouzdřuje data a~operace jedné funkční domény
(obrázek~\ref{fig:class-diagram}).

\texttt{SleepStoreContext} spravuje kolekci spánkových záznamů
a~zpřístupňuje operace pro jejich vytváření, editaci a~mazání
včetně detekce probíhající spánkové relace (\texttt{ongoingSleepSession}).

\texttt{NutritionStoreContext} uchovává jídelní záznamy ve struktuře
jako dvojici objektu \texttt{userMeal} a~pole \texttt{userFoods}; nad oběma
částmi poskytuje operace CRUD pro jídla a~jídelní deníky.

\texttt{WorkoutStoreContext} spravuje zaznamenané tréninky, šablony
tréninků a~referenční číselníky (cviky, styly a~typy sérií). Jeho
rozhraní je nejrozsáhlejší a~odráží komplexnost tréninkové domény.

\texttt{UserStoreContext} poskytuje přístup k~uživatelskému profilu,
stanoveným cílům a~protokolárním záznamům tělesných metrik
(hmotnost, výška).

Všechny čtyři providery jsou kompozičně zahrnuty v~komponentě
\texttt{StoreProvider}, která je vložena do kořene stromu aplikace.
Pro zpětnou kompatibilitu a~pohodlné použití v~místech, kde je
potřeba přistupovat k~více doménám současně, je k~dispozici fasádní
hook \texttt{useStore()}, který sloučí výstupy všech čtyř kontextů
do jediné struktury \texttt{StoreContextType}. Jeho použití však
způsobuje překreslení komponenty při změně v~\textit{jakékoliv}
doméně; z~toho důvodu je doporučeno v~nových komponentách
preferovat granulární hooky \texttt{useSleepStore()},
\texttt{useNutritionStore()}, \texttt{useWorkoutStore()}
a~\texttt{useUserStore()}.

\subsection{Výpočet metabolismu a denního energetického výdeje}

Bazální metabolismus (\textit{Basal Metabolic Rate}, BMR) vyjadřuje
minimální množství energie potřebné k~udržení základních životních funkcí
organismu v~klidovém stavu. V~aplikaci LifeMeter je BMR vypočítán rovnicí
Mifflin--St.~Jeor~\cite{mifflin1990bmr}, která je v~současné klinické
praxi považována za nejpřesnější predikci klidového energetického výdeje
pro obecnou populaci.

Pro pohlaví \textit{muž} platí:
\begin{equation}
  \text{BMR}_{\text{muž}} = 10 \cdot m + 6{,}25 \cdot h - 5 \cdot a + 5
  \label{eq:bmr-male}
\end{equation}

Pro pohlaví \textit{žena} platí:
\begin{equation}
  \text{BMR}_{\text{žena}} = 10 \cdot m + 6{,}25 \cdot h - 5 \cdot a - 161
  \label{eq:bmr-female}
\end{equation}

kde $m$ je tělesná hmotnost v~kilogramech, $h$ je výška v~centimetrech
a~$a$ je věk v~letech. Vstupní hodnoty zadané v~jednotkách lbs, st nebo
ft jsou před dosazením převedeny na soustavu SI.

Celkový denní energetický výdej (\textit{Total Daily Energy Expenditure},
TDEE) je získán vynásobením BMR koeficientem fyzické aktivity~$f$:

\begin{equation}
  \text{TDEE} = \text{BMR} \cdot f
  \label{eq:tdee}
\end{equation}

Koeficient~$f$ je uživatelem volen při onboardingu na základě úrovně
pohybové aktivity; každá úroveň je definována rozsahem
$[f_{\min},\, f_{\max}]$ a~ve výpočtu je použita střední hodnota
$f = (f_{\min} + f_{\max}) / 2$.

\subsection{Stanovení cílového denního kalorického příjmu}

Pokud uživatel v~průvodci nastavením zadá cílovou hmotnost a~cílové datum,
je vypočítán upravený denní kalorický příjem tak, aby bylo cíle dosaženo
v~požadovaném čase. Výpočet vychází z~energetické ekvivalence tukové
tkáně, kde změna $1\,\text{kg}$ tělesné hmotnosti odpovídá přibližně
$7\,700\,\text{kcal}$~\cite{hall2008energy}.

Celková kalorická bilance potřebná ke změně hmotnosti o~$\Delta m$
kilogramů za $d$~dní:
\begin{equation}
  \Delta E = \Delta m \cdot 7700 \quad [\text{kcal}]
  \label{eq:delta-e}
\end{equation}

Denní kalorická korekce:
\begin{equation}
  \delta = \frac{\Delta E}{d}
  \label{eq:daily-delta}
\end{equation}

Cílový denní příjem:
\begin{equation}
  C_{\text{cíl}} = \text{TDEE} + \delta
  \label{eq:calorie-goal}
\end{equation}

Hodnota $\delta > 0$ odpovídá kalorickému přebytku (nabírání hmotnosti),
$\delta < 0$ kalorickému deficitu (hubnutí). Pokud uživatel cílovou
hmotnost nezadá, je jako cílový příjem použito přímo~TDEE.

\subsection{Výpočet nutričních hodnot jídla}

Každá potravina v~databázi USDA FoodData Central~\cite{usda2024} obsahuje
nutriční hodnoty vztažené na $100\,\text{g}$ jedlého podílu. Při záznamu
jídla uživatel zadá skutečnou gramáž porce~$g$; obsah živiny~$N$ pro
danou potravinu je pak škálován proporčním faktorem:

\begin{equation}
  N_{\text{porce}} = N_{100} \cdot \frac{g}{100}
  \label{eq:nutrient-scale}
\end{equation}

Celkový kalorický obsah jídla je stanoven na základě Atwaterových
energetických koeficientů~\cite{atwater1902}:

\begin{equation}
  E = 4 \cdot P + 4 \cdot S + 9 \cdot T \quad [\text{kcal}]
  \label{eq:atwater}
\end{equation}

kde $P$ je obsah bílkovin, $S$ obsah sacharidů a~$T$ obsah tuků v~gramech.
Živiny jsou identifikovány prostřednictvím standardizovaných kódů USDA:
nutrient\_nbr 208 (energie), 203 (bílkoviny), 205 (sacharidy), 204 (tuky).

Pokud uživatel nestanoví explicitní cíle pro jednotlivé makronutrienty,
jsou tyto automaticky odvozeny z~TDEE s~výchozím rozložením 30\,\%
bílkoviny, 40\,\% sacharidy a~30\,\% tuky:

\begin{align}
  P_{\text{cíl}} & = \left\lfloor \frac{\text{TDEE} \cdot 0{,}30}{4} \right\rfloor \label{eq:macro-p} \\[4pt]
  S_{\text{cíl}} & = \left\lfloor \frac{\text{TDEE} \cdot 0{,}40}{4} \right\rfloor \label{eq:macro-s} \\[4pt]
  T_{\text{cíl}} & = \left\lfloor \frac{\text{TDEE} \cdot 0{,}30}{9} \right\rfloor \label{eq:macro-t}
\end{align}

\subsection{Fulltextové vyhledávání potravin}

Databáze potravin čítá přes milion záznamů z~datového zdroje USDA
FoodData Central~\cite{usda2024}. Efektivní vyhledávání je zajištěno
kombinací PostgreSQL full-text search a~GIN indexu. Sloupec
\texttt{search\_vector} je persistován jako generovaný sloupcový
\texttt{tsvector}:

\begin{lstlisting}[language=SQL, caption={Definice generovaného sloupce a GIN indexu},
  label={lst:tsvector}]
ALTER TABLE food
  ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (to_tsvector('english', description)) STORED;

CREATE INDEX idx_food_search_vector
  ON food USING GIN (search_vector);
\end{lstlisting}

Vstupní dotaz je na straně serveru sanitizován -- jsou odstraněny speciální
znaky operátorů PostgreSQL (\texttt{\&}, \texttt{|}, \texttt{!}, závorky
apod.) a~každý token je rozšířen o~prefixový operátor~\texttt{:*},
který zajišťuje, že dotaz \uv{appl} nalezne i~záznamy obsahující
\uv{apple} nebo \uv{applesauce}. Jednotlivé tokeny jsou spojeny
operátorem AND:

\begin{lstlisting}[caption={Transformace vstupního řetězce na tsquery},
  label={lst:query-transform}]
vstup:  "whole grain bread"
vystup: "whole:* & grain:* & bread:*"
\end{lstlisting}

Výsledky jsou řazeny víceúrovňovým klíčem tak, aby byly upřednostněny
stručné a~přesné shody před delšími nebo méně relevantními záznamy:

\begin{enumerate}
  \item přesná shoda s~dotazem (\texttt{ILIKE \$query}) -- priorita~0,
  \item prefixová shoda (\texttt{ILIKE \$query||'\%'}) -- priorita~1,
  \item ostatní výsledky seřazené dle \texttt{ts\_rank\_cd}
        a~délky popisu -- priorita~2.
\end{enumerate}

\subsection{Offline fronta požadavků}
\label{sec:offline-fronta}

Aplikace implementuje princip \textit{offline-first}~\cite{offlinefirst2015}:
při ztrátě síťového připojení nejsou mutační požadavky (POST, PUT,
DELETE) zahozeny, nýbrž jsou přidány do fronty v~lokálním
úložišti MMKV~\cite{mmkv2024} a~odeslány po obnovení připojení.
Čtecí požadavky (GET, HEAD) jsou vždy odesílány okamžitě a~do fronty
zařazeny nejsou.

\begin{lstlisting}[caption={Pseudokód zpracování HTTP požadavku s offline frontou},
  label={lst:offline-queue}]
FUNCTION request(path, init) -> T | undefined
  IF isOffline AND method(init) != GET AND offlineModeEnabled THEN
    IF NOT queue.contains(path, init) THEN   -- deduplikace
      queue.push(path, init)                 -- persistuj do MMKV
    END IF
    RETURN undefined                         -- optimisticky navrat
  END IF

  controller <- AbortController(timeout = 5000 ms)
  response   <- HTTP.send(path, init, signal = controller)
  RETURN parse(response)
END FUNCTION

FUNCTION flushQueue() -> void
  pending <- queue.readAll()
  queue.clear()                              -- smaz pred odesilani

  FOR EACH (path, init) IN pending DO
    IF request(path, init) FAILS THEN
      queue.push(path, init)                 -- vrat zpet pri selhani
    END IF
  END FOR
END FUNCTION
\end{lstlisting}

Deduplikace je zajištěna porovnáním serializované cesty a~těla požadavku
před vložením do fronty, čímž je zamezeno opakovanému záznamu totožné
operace.

Každý HTTP požadavek je opatřen timeoutem $5\,000\,\text{ms}$
prostřednictvím \texttt{Abort}\allowbreak\texttt{Controller}, čímž je zamezeno blokování
při nedostupném serveru.
